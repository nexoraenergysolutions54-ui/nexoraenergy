# Repository Guidelines

## Project Structure & Module Organization

This is a TanStack Start + Vite React/TypeScript site for Nexora. Application code lives in `src/`. File-based routes are in `src/routes/`; each `.tsx` file maps to a URL, and `src/routes/__root.tsx` is the shared app shell. Shared layout and feature components live in `src/components/`, with reusable Radix-style primitives in `src/components/ui/`. Utilities and server-only helpers are in `src/lib/`, hooks in `src/hooks/`, and images in `src/assets/`. Do not edit `src/routeTree.gen.ts` by hand; it is generated by TanStack Router.

## Build, Test, and Development Commands

- `npm run dev`: start the local Vite development server.
- `npm run build`: create a production build in `dist/`.
- `npm run build:dev`: build using development mode settings.
- `npm run preview`: serve the built app locally for verification.
- `npm run lint`: run ESLint and Prettier checks across the repo.
- `npm run format`: format files with Prettier.

The repo includes both `package-lock.json` and `bun.lock`; use the package manager already expected by the active workflow, and avoid mixing lockfile updates in unrelated changes.

## Coding Style & Naming Conventions

Use TypeScript and React function components. Prettier is authoritative: 100-character print width, semicolons, double quotes, and trailing commas. Prefer imports through the `@/*` alias for source files, for example `@/components/Layout`. Component files use PascalCase, such as `Navbar.tsx`; hooks use `use-*.tsx`; route files use URL-oriented lowercase names such as `about.tsx` and `services.tsx`.

## Testing Guidelines

No automated test command is currently defined in `package.json`. For now, validate changes with `npm run lint`, `npm run build`, and manual browser checks of affected routes. If tests are added, place them near the code they cover or in a clear `src/__tests__/` structure, and add the test command to `package.json`.

## Commit & Pull Request Guidelines

Recent history uses short subjects such as `Built Nexora enterprise pages` and `Changes`. Prefer concise, descriptive imperative subjects that explain the user-visible change, for example `Add careers page content`. Pull requests should include a summary, affected routes or components, verification steps, and screenshots for visual changes.

## Security & Configuration Tips

Do not commit secrets, local logs, or generated build output. Keep server-only code in `*.server.ts` modules or use TanStack Start server-only conventions; do not import Next.js `server-only`.
